## 4. Architecture decisions (agreed with the user — do not re-litigate)

### 4.1 Three layers, one-way dependencies

| Layer | Owns | Scope | Runtime lookup |
|---|---|---|---|
| **Theme** | `GameUITheme.tres` — StyleBoxes, fonts, spacing, palette | Global, static | No |
| **Cosmetic catalog** | id → asset manifest of every variant | Global, static, shared client+server | No |
| **Equipped state** | which player wears which cosmetic id | Per-player, runtime | Yes |

**The Theme must never know a player exists. The cosmetic registry must never
read the Theme.** They meet only in the scene tree, never in code.

**Cosmetics are opaque to the Theme.** A global re-theme must not re-tint a
cosmetic. If the theme could recolor cosmetics, every new theme would have to be
visually validated against every cosmetic variant — an N×M problem. Static
chrome re-themes; cosmetics do not.

### 4.2 Identity model — forward-compatible with a future database

Split the runtime player from the durable profile:

- **Runtime player** — `ws`, `score`, `blocks`, `lastPlacementTime`. Dies with
  the room. Unchanged from today.
- **Profile** — `displayName`, `avatarId`, `equipped{}`, `owned[]`. Keyed by a
  `profileId` that is **not** the ephemeral `P1` room id.

Add `src/Server/app/Profile_Store.js` following the **same adapter shape as
`src/Server/app/Redis_State.js`** — a defined interface with an in-memory
implementation now, a real database implementation later. Callers never change
when the backing changes.

`profileId` before accounts: the Godot client generates a UUID once and persists
it locally (`user://`), sending it on connect. This gives real continuity — a
player keeps their name and avatar across sessions on the same device — and when
accounts arrive you migrate device-id → account-id with nothing above the store
changing.

**The client sends a `profileId`, never profile data.** The server resolves
id → profile and broadcasts the result. This is what makes the wire contract,
the client render path, and the fallback final on day one.

> **Security note to carry forward:** a device id is trivially spoofable. It is
> fine as a cosmetics key, but it must **not** become the auth boundary when
> ownership becomes purchasable. When real accounts land, the device id is
> demoted to a migration hint, not a credential.

### 4.3 Why names must come from the server even while fake

A client-side placeholder name is not a placeholder, it is a bug: three clients
each inventing their own names means each player sees different names for the
same people. In a game whose tension depends on knowing who placed what, that is
broken. Names are server-assigned and broadcast, always. Bots get names too
(`isBot` is already on the wire).

### 4.4 Cosmetic slots and the gameplay-signal constraint

Four slots: **avatar, brick, chat bubble, impact bar.**

Brick colour identifies the player who placed it (confirmed: the five brick
colours in `148:3444` match the five impact-track dot colours, and
`PlayerColors.gd` + Tower Stack already work this way). Impact-track dots carry
the same signal.

> **Catalog rule — must be written into the catalog spec so the artist knows it
> before drawing variants:** a brick or impact-bar cosmetic may change material,
> pattern, outline, and corner treatment. It may **not** change hue, and it may
> not override face-expression semantics. Avatar and chat-bubble cosmetics have
> no such constraint.

Player colour stays **seat-assigned, never player-chosen** — two players picking
the same colour would destroy the readability signal.

### 4.5 Equipping is out-of-match only

Equipped state is static for a room's lifetime, so it rides the
`room_created` / `room_resumed` roster, **not** the per-tick `game_state`
broadcast. Do not add cosmetic fields to the per-tick payload. This eliminates
mid-match sync, late-join divergence, and interaction with Impact rollback.

## 5. Confirmed UI behaviour

- **Popovers** (Team Inventory, Quick Chat, Power, Quest) are four instances of
  **one** `PopoverPanel` component — title, rule, list of rows. They differ only
  in anchor and row content. Build it once.
- Popovers **auto-close after 4s (default, configurable)**, on **tap outside**,
  and on **drag start**. Dismiss-on-drag-start matters: in a 30-second round a
  panel that swallows the first drag costs a placement.
- Popovers are **non-blocking** — the round is real-time, there is no pause.
- **Round Time** is placeholder art; follow prototype logic. `secondsRemaining`
  already drives it — no server change.
- **Tower stability**: defer the brick face-expression idea. Follow the existing
  prototype behaviour where bricks tilt left or right.
- **Score popups**: design is WIP. For now render the popup **to the right of
  the player's avatar** in the left rail.
- **Level summary**: design is WIP. For now render a **placeholder popup
  centered on screen**, as the prototype does.
- **Hamburger menu** (`top-bar`): tappable, does nothing.
- **Join button** (Join Server screen): tappable, does nothing. The "Player
  name" field is for private servers, which are WIP — **ignore it entirely for
  now.** Names remain server-assigned.
- **Find Match** behaves exactly like the prototype's Connect button. If bots
  are enabled it proceeds to Play; otherwise it waits for 3 players.
- **Cancel** (Find Match screen) → leaves the queue → returns to Join Server.
- **`room_closed`** → returns to Join Server.
- **Debug button**: draggable overlay, persists across all screens, **position
  resets on every room/launch** (do not persist to disk). Tap opens the existing
  debug menu unchanged.
- Room size stays **3 players**. The artist sized the rail for 5 as future
  headroom and will hide seats 4–5 and their impact entries.

## 6. Phases

Each phase ends with a verification step and a stop for user confirmation.

### Phase 1 — Theme extraction and palette derivation (Completed)

**Intent: zero visual change.** This is a pure refactor that makes every later
phase cheap. Do not apply new design values in this phase.

1. Derive the palette and type scale from Figma node `34:3` and its children by
   reading raw fills (there are no variables). Produce a short palette summary
   **in chat** — named tokens, hex values, and where each is used — and get the
   user to confirm intent before applying. Do not write this to `docs/`.
2. Set the base viewport in `project.godot`: 412×917, portrait orientation,
   stretch mode `canvas_items` with an appropriate aspect setting for a range of
   Android devices.
3. Move existing inline `theme_override_*` properties from nodes in
   `Cor/Scenes/GameUI.tscn` into `Cor/Themes/GameUITheme.tres` as real Theme
   entries (StyleBoxFlat, font sizes, colors, constants).

**Verify:** launch the app and confirm it is visually identical to before, and
that `Tests/CiSmokeTest.gd` still passes. Any visual difference in this phase is
a regression, not progress.

### Phase 2 — Screen navigation (Completed)

1. Add `ScreenManager` owning which screen scene is active. `Main.tscn` becomes
   the host; `GameUI.tscn` stops being statically instanced.
2. New scenes + scripts: `JoinScreen` (`87:852`), `FindMatchScreen` (`90:994`).
   Build **structure and layout only** — these are wireframes. Let them inherit
   the Phase 1 Theme so they look consistent without inventing new styling.
3. Move the debug button out of `GameUI.tscn` to a root-level overlay owned by
   `ScreenManager`, above whatever screen is active. Make it draggable.
   Position resets on launch/room.
4. Add a `leave_queue` client→server message and handle it in
   `src/Server/app/Lobby_Manager.js`. Prefer this over relying on socket close,
   which leaves a session in Redis with its TTL and makes a relaunch try to
   resume a queue the player deliberately left.
5. Wire the flow: Join → Find Match → Play; Cancel → Join; `room_closed` → Join.

> **Do not refactor `Main.gd` in this phase.** Extract only the screen-switching
> seam and leave the gameplay rendering internals in place. Decomposing a
> 2227-line untested file at the same time as restructuring navigation means a
> break cannot be attributed. Decomposition is a separate future task with its
> own verification.

**Verify:** full flow by hand — launch → Join → Find Match → bots enabled →
Play → force room close → back to Join. Cancel from Find Match and confirm the
server dequeues (check logs). Smoke test passes.

### Phase 3 — Player identity

1. Add `src/Server/app/Profile_Store.js` mirroring `Redis_State.js`'s adapter
   shape. In-memory implementation returning generated defaults.
2. Client generates and persists a `profileId` UUID at `user://`, sends it on
   connect. `Sys/NetMan/NetworkManager.gd`.
3. Server resolves `profileId` → profile, assigns `displayName` (word list or
   `Player N`) and `avatarId` (deterministic from seat index) for unknown ids,
   including bots.
4. Add `displayName` and `avatarId` to the `room_created` / `room_resumed`
   roster. **Not** to the per-tick `game_state` payload.
5. `avatarId` is a **catalog id**, not a texture path — this makes the avatar
   the first cosmetic slot and lets Phase 5 generalize rather than invent.

**Verify:** run a 3-player room (bots enabled), confirm all three clients render
the same names for the same players, and that names survive a reconnect.

### Phase 4 — Play screen reskin

Import art assets. Folder convention:

  procedural: fills, rounded rects, borders, gradients, shadows, type, spacing.
- `Cor/Art/Static/` — imported images for genuinely painted art: backgrounds,
  logo, decorative frames, custom icon glyphs.
- `Cor/Art/Cosmetics/<slot>/<variant>/` — per-variant cosmetic art (Phase 5).

Painted buttons that need to stretch use `StyleBoxTexture` with 9-slice.

Score popups → right of the player avatar. Level summary → centered placeholder.

**Verify:** play a full level with bots against the Figma frames side by side.
Confirm popovers open/close per §5, drag-to-place still works, and score events
and level summaries still render.

### Phase 5 — Cosmetic scaffolding

1. Generalize `Cor/Scripts/PlayerColors.gd`'s pattern — it is already
   "player id → visual, with fallback" — into a `CosmeticRegistry` with
   `resolve(player_id, slot) -> Resource` and a per-slot default fallback.
2. Visual nodes ask the registry and nothing else. **No node reaches into
   network state directly.** Bots and unresolvable ids hit the fallback path for
   free, which is most of the correctness story.
3. Slots: avatar (already live from Phase 3), brick, chat bubble, impact bar —
   under the §4.4 hue constraint.
4. Add the shared cosmetic catalog, readable by both client and server.
5. Server validates `equip` requests as `id ∈ catalog && id ∈ owned`. Ownership
   is stubbed to "everyone owns everything" in this phase.

**Verify:** force-equip non-default variants for one player, confirm all three
clients render that player's cosmetics identically, and that an unknown
cosmetic id falls back to default instead of erroring.
