## 2. Environment facts (why the editor can't reproduce)

`src/Client/App/corp-tower/project.godot`:
- `[display] window/size/viewport_width=412`, `viewport_height=917`
- `window/stretch/mode="canvas_items"`, `window/stretch/aspect="expand"`,
  `window/stretch/aspect.web="keep"`, `window/handheld/orientation=1`
- `[input_devices] pointing/emulate_mouse_from_touch=true` (added last session)

Consequences:
- The **editor runs at exactly 412×917** — the one size where the authored
  popover layout is already correct. `expand` (Android) grows the viewport past
  412×917; `keep` (web) pillarboxes to 412×917. See
  `docs/components/Godot Client App.md` for why the two aspects only coincide at
  412×917 and the editor cannot validate device layout.
- Both `emulate_mouse_from_touch` and `emulate_touch_from_mouse` are effectively
  on (the latter is Godot's default and is not overridden), so **every physical
  tap produces two `_input` events**: a real one and an emulated partner.

## 3. Mechanism map (files + functions as they stand now)

Input path (a tap → a popover):
- `Cor/Scripts/Main.gd` `_input()` — calls `inventory.handle_input(event)`,
  `power.handle_input(event)`, then `trigger_router.process(event)`; on a true
  return it calls `get_viewport().set_input_as_handled()`.
- `Cor/Scripts/GameUi/PointerTriggerRouter.gd` `process()` — **mouse-only**: it
  reacts only to `InputEventMouseButton` left-press, runs guards (debug overlay
  open, level-summary visible), then hit-tests the four trigger `get_global_rect()`s
  in order and calls the matching opener. It returns `true` (→ consumed) only for
  a mouse press that hits a trigger. **It never consumes touch events.**
- Openers: `Main.gd open_team_inventory_popover()`,
  `GameUi/QuickChatController.gd open_quick_chat_popover()`,
  `GameUi/PowerController.gd open_power_popover()`,
  `GameUi/QuestController.gd on_quest_chip_pressed()`/`open_quest_popover()`.
- `GameUi/PopoverCoordinator.gd` — `present()` / `is_open()` / `close_active()`;
  toggling is guarded by the popover's live `.visible`.

Why triggers use `_input` at all: in `Cor/Scenes/GameUI.tscn` the two popover
instances (`TeamInventoryPopover`, `QuestPopover`) are declared **after**
`PlayField` (which holds the triggers). Each popover's `OutsideCatcher` is a
full-screen `Control` with `mouse_filter = STOP`, so while a popover is open it
outranks the triggers in GUI hit-testing. `_input` runs before GUI routing, so
hit-testing there lets a tap reach a trigger regardless of z-order.

Popover close-on-outside:
- `Cor/Scripts/PopoverPanel.gd` `_on_outside_catcher_gui_input()` closes the
  popover on **either** `InputEventMouseButton` **or** `InputEventScreenTouch`
  press.

Popover positioning:
- `PopoverPanel.gd` `get_card_size()` (content size via
  `get_combined_minimum_size()`) and `set_card_global_position()` (resets the
  Card's anchors to top-left, sets explicit size, clamps into the popover rect).
- Callers: `Main.gd position_shared_popover_card()` (anchors the shared card to
  `power.power_trigger` / `chat.quick_chat_trigger` / `team_inventory_button`)
  and `QuestController.gd position_quest_popover_card()` (anchors to the quest
  chip). Both compute the target from the trigger's live `get_global_rect()`.

Scene/scale context:
- `Cor/Scripts/ScreenManager.gd` instances `GameUI.tscn` into `ScreenContainer`
  (a plain full-rect `Control` in `Main.tscn`) — **no app-level scale applied**.
- **Diagnostic clue:** ScreenManager's own floating **DebugButton** is a normal
  `Control` using `gui_input`/tap, and it is believed to work on Android. If so,
  ordinary GUI input works on device and the trigger problem is specific to the
  `_input` + OutsideCatcher interaction, not input delivery in general. **Verify
  this clue first** — it strongly narrows the search.

## 4. What was already tried (do NOT just repeat these)

Two prior attempts, both verified only in-editor/gdparse (no device run by the
author), both insufficient:

1. **Symmetric frame-dedup** in the trigger `_input`: stamp
   `last_pointer_trigger_frame` in whichever of the mouse/touch branches fired
   first, skip the other if same frame. Failed because the emulated partner can
   arrive on a *different* frame on web, re-triggering.
2. **Mouse-only trigger activation** (current state): ignore touch in the router,
   rely on `emulate_mouse_from_touch`. Plus runtime card positioning
   (`set_card_global_position`) and content-based `get_card_size()`. This fixed
   the editor + the GUT layout test but **did not fix the device symptom.**

Lesson: stop reasoning purely from the code; get device telemetry first.

## 5. Diagnostic protocol (do this FIRST, on a real build)

Add a **temporary** on-screen readout (e.g. a `Label` in the HUD, or reuse
`StatusLabel`) — do not rely on `print()` alone on web/Android. Log, on every
trigger tap and every popover open:

1. `get_viewport().get_visible_rect().size` — is the device viewport actually
   412×917, or larger (Android `expand`) / offset?
2. For the tapped trigger: `get_global_rect()`.
3. The popover root's `get_global_rect()` and the Card's `get_global_rect()` +
   `card.get_combined_minimum_size()` immediately after positioning.
4. Every `InputEvent` that reaches `Main._input`: its class name, `position` /
   `global_position`, and whether `trigger_router.process` consumed it.
5. In `PopoverPanel._on_outside_catcher_gui_input` and `close()`: log the event
   class and the time/frame, so you can see **whether the popover is being closed
   on the same tap that opened it.**

Then reproduce each symptom once and read the log. This single pass should
distinguish the hypotheses below.

## 6. Leading hypotheses (ranked) + candidate fixes

### Bug B (trigger tap "does nothing") — most likely a same-tap self-close

**Hypothesis B1 (highest confidence).** The router consumes only the *mouse*
event of a tap. The emulated *touch* partner of the same physical tap is left
unconsumed, propagates into GUI routing, and the **freshly-opened popover's
`OutsideCatcher` receives it and closes the popover on the same tap** — net
"nothing happens." `_on_outside_catcher_gui_input` closes on touch as well as
mouse (confirmed), so this fires wherever a tap yields a mouse + a stray touch.

*Confirm with §5 step 5 (does close() fire immediately after open on one tap?).*

Candidate fixes (pick after confirming; each is small):
- Have the `OutsideCatcher` **ignore the opening tap** — e.g. the popover records
  the frame/time it opened and `_on_outside_catcher_gui_input` ignores presses
  within the same input frame / a few ms of `open()`.
- Or make trigger activation **consume the whole tap**: when the router opens a
  popover, also swallow the emulated partner (e.g. the opener/`_input` consumes
  both the mouse and the same-position touch this frame), so nothing reaches the
  OutsideCatcher.
- Or reconsider the architecture: connect the trigger buttons' **native
  `pressed` signals** and only fall back to `_input` interception while a popover
  is already open (needed just for switching between popovers). If the DebugButton
  clue (§3) holds, native button input works on device and this removes the
  OutsideCatcher race for the common "open from closed" case.

**Hypothesis B2.** On Android the emulated mouse event is not delivered as
expected (so mouse-only never fires). *Confirm with §5 step 4 — is there an
`InputEventMouseButton` at all on an Android tap?* If not, revert to handling
touch directly but solve the double-fire via B1's self-close fix rather than
frame-dedup.

**Hypothesis B3.** A guard is wrong on device (e.g. `debug_panel.is_open()` or
`summary.is_overlay_visible()` returning true spuriously), blocking activation.
*Confirm with §5 step 4 — does `process` reach `try_activate` and return false?*

### Bug A (popover card mis-positioned/stretched)

**Hypothesis A1.** The Card's size or the trigger rects read in a coordinate
space that doesn't match where the Card is drawn on device (viewport ≠ 412×917
under `expand`, or a content-scale/canvas transform offset). *Confirm with §5
steps 1–3 — compare the logged viewport size and rects to 412×917.* If the
viewport is larger, the trigger `get_global_rect()` and popover rect should still
be consistent (both in the same space) and the card should track — so if it does
NOT track, look for a transform mismatch between `get_global_rect()` and the
Card's parent space.

**Hypothesis A2 (stretch persists).** `set_card_global_position` resets anchors
to 0 and sets `card.size`, but the Card is a `PanelContainer`; if anything
re-runs container layout (or the anchors don't actually take on device), the card
re-stretches. *Confirm with §5 step 3 — is `card.get_global_rect().size` the
content size or full-screen?* If it re-stretches, set the card's
`custom_minimum_size`/`size` explicitly and/or verify the anchor reset holds
across a frame (position deferred to after the next layout with `await
get_tree().process_frame` before measuring).

**Hypothesis A3.** `get_combined_minimum_size()` returns a bad (huge) value on
device with the real theme/fonts, so the card is mis-sized. *Confirm with §5
step 3.* If so, drive the size from a fixed design size or the authored Card
offsets rather than content.
